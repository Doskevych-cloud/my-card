# UI-правила (обов'язкові)

Живі правила щодо уніфікації UX на всіх сторінках застосунку RE:ACT.
Кожен новий звіт / сторінка / вкладка мусить їх виконувати.

---

## 1. Org-фільтр (Всі / Реакт / Генерація)

**Правило:** На будь-якій сторінці, де є фільтр по організації,
чіпи йдуть у фіксованому порядку:

```html
<span class="group">
  <span class="chip" data-org="" onclick="setOrg('')">Всі</span>
  <span class="chip active" data-org="Реакт" onclick="setOrg('Реакт')">Реакт</span>
  <span class="chip" data-org="Генерація" onclick="setOrg('Генерація')">Генерація</span>
</span>
```

Вимоги:
- **Завжди три варіанти**: Всі · Реакт · Генерація. Порядок саме такий.
- Кнопка називається **«Всі»** (не «Усі», не «All», не «Обидві»).
- `data-org=""` для «Всі» — порожній рядок, бекенд цим сигналом повертає
  усі орг.
- Якщо користувачу доступна лише одна орг — бекенд permissions сам ховає
  недоступні чіпи (`display:none`); логіку не змінювати.

### Поведінка при `state.org === ''` (режим «Всі»)

- **Сторінки Sales / OSP / Warehouses / Reports**: клієнти/менеджери/
  товари з різних організацій мерджаться за ідентифікатором (name / ref),
  суми зваловуються, списки (orders, reserves) конкатенуються.
- **ar_aging.html / finance**: backend повертає агреговану відповідь
  без розбивки по орг при `org=''` — фронт рендерить як звичайно.

Реалізація merge — окрема функція `getManagers()` / `getClients()` /
`getCombinedItems()` поряд з `render()`. Не вбудовувати merge у render.

## 2. Обов'язкові UI-елементи на звітах

- **Page tabs** (`.page-tabs` / `.ptab`) — перемикач між спорідненими
  звітами (Sales / OSP / AR Aging тощо).
- **Toolbar** (`.toolbar`) — єдиний контейнер із:
  - датою (або періодом)
  - пресетами (Цей місяць / Минулий / Квартал / Рік)
  - org-чіпами (див. §1)
  - сортуванням, якщо є
  - пошуком
- **AppHeader** монтується *до* `Auth.require()` (eager mount), щоб
  кнопка теми працювала навіть у preview-режимі / без сесії.

---

## 3. Кольори

Використовувати **виключно** CSS-змінні з палітри фінансів:
`--text`, `--text2`, `--text3`, `--accent`, `--red`, `--green`, `--blue`,
`--yellow`, `--bg`, `--bg2`, `--bg3`, `--surface`, `--surface2`,
`--border`, `--border2`.

Для aging-бакетів — фіксоване відповідення:
- `d7` (До 7) — `--green`
- `d14` (8–14) — `--blue`
- `d30` (15–30) — `--yellow`
- `d60` (31–60) — `--accent`
- `d61p` (Від 61) — `--red`

---

## 4. Grand-total в доларах

На всіх звітах з мульти-валютними сумами **загальна цифра — в USD**,
конвертація через `fetchFxOnDate(KV, onDate)` на бекенді. Розбивку
по оригінальних валютах не показуємо в головній плашці (якщо
потрібно — окремою дрібною підстрокою, але дефолт — один USD-рядок).

---

## 5. Картки «Без менеджера» / «Unassigned»

Якщо бекенд агрегує щось без власника (менеджер, manager_key, тощо),
картка показується **як звичайна** — без червоних рамок, без
попереджень, без емодзі. Факт наявності — вже сигнал.
